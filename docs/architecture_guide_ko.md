# 📘 TaskFlow 프로젝트 기술 명세 및 아키텍처 가이드 (Detailed)

이 문서는 TaskFlow To-Do 애플리케이션의 **프론트엔드, 백엔드, 데이터베이스 설계, 클라우드 배포 파이프라인, 그리고 향후 보안 아키텍처**까지 프로젝트의 모든 기술적 세부 사항을 총망라한 상세 가이드입니다.

---

## 1. 🏗️ 전체 기술 스택 및 구조 (Tech Stack)

TaskFlow는 최신 웹 개발 트렌드를 반영하여 성능, 유지보수성, 배포 편의성을 극대화한 풀스택(Full-Stack) 아키텍처로 설계되었습니다.

| 영역 | 기술 및 도구 | 역할 및 선택 이유 |
|---|---|---|
| **Frontend** | React 19, Vite 8 | 빠르고 즉각적인 UI 렌더링 및 초고속 로컬 빌드 환경 구축 |
| **Styling** | Vanilla CSS | 외부 라이브러리(Tailwind 등) 의존성 없이 순수 CSS 변수(Variables)를 활용한 커스텀 디자인 시스템 구축 |
| **Backend** | .NET 9.0 (ASP.NET Core API) | 강력한 타입 안정성, 고성능 멀티스레딩, 안전한 REST API 제공 |
| **ORM** | Entity Framework Core (EF Core) | 객체 지향적인 C# 코드를 데이터베이스 쿼리로 안전하게 자동 변환 |
| **Database** | SQLite | 별도의 서버 설치가 필요 없는 파일 기반 DB로, 소규모~중규모 개인 앱에 최적화 |
| **Container** | Docker | 프론트와 백엔드를 하나의 환경(컨테이너)으로 묶어 어디서든 동일하게 실행되도록 보장 |
| **Cloud** | Fly.io | 작성된 Docker 컨테이너를 전 세계 어디서든 접속 가능하게 띄워주는 클라우드 플랫폼 |
| **CI/CD** | GitHub Actions | 코드가 수정(Push)될 때마다 사람의 개입 없이 자동으로 서버를 최신화하는 봇 |

## 2. 🎨 프론트엔드 아키텍처 (React & Vite)

프론트엔드는 사용자의 쾌적한 경험(UX)과 코드의 직관성에 초점을 맞췄습니다.

### 2.1. 관심사의 분리 (API Service Layer)
UI를 그리는 코드(`App.jsx`)와 서버와 통신하는 코드(`todoApi.js`)를 엄격하게 분리했습니다.
* **`todoApi.js`**: `fetch` 함수를 이용해 백엔드와 통신하며, 에러가 발생하면 일관된 방식으로 에러를 가공하여 UI 쪽에 던져줍니다.
* 이를 통해 UI 컴포넌트는 "데이터를 어떻게 가져올까?" 고민할 필요 없이 "데이터를 어떻게 예쁘게 보여줄까?"에만 집중할 수 있습니다.

### 2.2. 상태 관리와 Optimistic Update (낙관적 업데이트)
React의 상태(State) 관리를 통해, 사용자가 할 일을 추가하거나 완료 체크를 하면 **새로고침 없이 화면이 즉시 부드럽게 변경**됩니다.
어떤 변경(Create/Update/Delete)이 일어나면 내부적으로 백엔드에 요청을 보낸 뒤, 성공하면 즉시 `fetchTodos()`를 다시 호출하여 서버와 화면의 데이터를 동기화합니다.

### 2.3. 순수 CSS (Vanilla CSS) 디자인 시스템
무거운 외부 UI 라이브러리를 쓰지 않고, `index.css`에 **디자인 토큰(CSS 변수)** 을 정의하여 사용했습니다.
* `--bg-primary`, `--text-primary`, `--accent-color` 등 색상과 간격 규칙을 미리 정해두어, 앱 전체적으로 완벽하게 통일된 다크 모드(Dark Mode) 디자인을 일관되게 유지합니다.

## 3. ⚙️ 백엔드 아키텍처 (ASP.NET Core Web API)

백엔드는 외부의 악의적인 요청으로부터 시스템을 보호하고, 데이터를 안전하게 가공하는 '수문장' 역할을 합니다.

### 3.1. DTO (Data Transfer Object) 패턴
데이터베이스와 1:1로 연결된 실제 모델(`TodoItem.cs`)을 사용자(브라우저)에게 절대 직접 노출하지 않습니다.
대신 **DTO(`CreateTodoDto`, `TodoResponseDto` 등)** 라는 '데이터 운반 전용 택배 상자'를 만들어서 통신합니다.
* **장점**: 사용자가 조작하면 안 되는 데이터(예: `CreatedAt` 등)를 원천적으로 차단하며, 필요한 데이터만 딱 맞게 가공해서 보낼 수 있습니다.

### 3.2. Data Annotations & 유효성 검사
사용자가 보낸 데이터가 올바른지 서버가 철저하게 검사합니다.
* 예: "제목(Title)은 필수 항목이며 최대 100자를 넘을 수 없다", "마감일(DueDate)이 잘못된 형식이다" 등.
* 이 검사를 통과하지 못하면 백엔드는 데이터베이스에 접근조차 하지 않고 즉시 에러(400 Bad Request)를 반환합니다.

### 3.3. 전역 예외 처리 (Global Exception Handling)
서버 내부에서 예상치 못한 에러가 터져도 서버가 죽거나 해커에게 서버의 내부 소스 코드가 노출되지 않도록, 모든 에러를 낚아채서 예쁜 JSON 형태의 에러 메시지로 감싸서 브라우저에 돌려보냅니다.

## 4. 🗄️ 데이터베이스 설계 (SQLite & EF Core)

### 4.1. 왜 SQLite를 썼나요?
보통 거대한 상업용 서비스는 MySQL이나 PostgreSQL 같은 무거운 DB 서버를 씁니다. 하지만 이 프로젝트는 개인 혹은 소수 인원이 사용하는 앱이므로, 파일 하나(`todos.db`)에 모든 게 저장되는 SQLite가 유지보수와 배포 관점에서 압도적으로 유리합니다.

### 4.2. 인덱스(Index) 최적화
빠른 검색을 위해 DB 설계 단계부터 고민이 들어갔습니다.
* 상태(Status)나 중요도(Priority), 마감일(DueDate) 기준으로 정렬하고 검색할 때 DB가 전체 데이터를 다 뒤지지 않도록 **인덱스(색인)** 를 걸어두어 조회 속도를 극대화했습니다.

## 5. 📦 도커(Docker) 아키텍처 (3-Stage Build)

우리의 `Dockerfile`은 예술적인 **3단계(3-Stage) 구조**로 짜여 있습니다.

1. **Stage 1 (프론트엔드 빌드)**: Node.js 환경에서 React 코드를 가져와 압축된 순수 HTML/JS/CSS 정적 파일로 변환합니다.
2. **Stage 2 (백엔드 빌드)**: .NET SDK 환경에서 C# 백엔드 코드를 가져와 고성능 실행 파일로 압축(Publish)합니다.
3. **Stage 3 (런타임 조립)**: 위 두 단계에서 사용된 무거운 빌드 도구들은 싹 다 버리고! 오직 가벼운 실행 환경(Alpine Linux)만 남긴 뒤, **압축된 프론트엔드 파일과 백엔드 실행 파일만 쏙 뽑아와서 하나의 가벼운 박스에 합칩니다.**
   - 프론트엔드 파일들은 백엔드의 `wwwroot` (정적 파일 서빙 폴더) 안에 들어가서, 백엔드 서버 하나가 프론트와 API를 동시에 제공하게 됩니다.

## 6. 🌐 배포 및 파이프라인 (Fly.io & GitHub Actions)

로컬에서 잘 돌아가는 코드를 24시간 멈추지 않는 글로벌 서버로 옮기는 과정입니다.

### 6.1. 무결점 데이터 보존 (Fly.io Volume Mount)
도커 컨테이너는 업데이트될 때 기존 박스를 부수고 새 박스를 띄웁니다. 파일 형태인 SQLite(`todos.db`)가 박스 안에 있으면 업데이트할 때마다 데이터가 다 날아가는 대참사가 발생합니다.
* **해결책**: Fly.io에 `taskflow_data`라는 영구적인 **가상 외장 하드디스크(Volume)** 를 만들었습니다.
* `fly.toml` 설정 파일에 **"서버가 새로 뜰 때마다 무조건 이 외장 하드를 서버 안의 `/data` 폴더에 꽂아라"** 라고 지시해 두어, 데이터가 영구적으로 안전하게 보존됩니다.

### 6.2. 100% 자동화된 CI/CD 파이프라인
더 이상 명령어를 직접 칠 필요가 없습니다. `.github/workflows/fly.yml` 파일이 다음을 자동화합니다.
1. 개발자가 `main` 브랜치에 코드를 **Push** 합니다.
2. 깃허브 서버가 변경을 감지하고, 빈 리눅스 컴퓨터를 하나 빌려서 최신 코드를 다운받습니다.
3. 깃허브 설정(Secrets)에 숨겨둔 `FLY_API_TOKEN` (우리가 발급받아 등록한 그 긴 암호)을 사용해 Fly.io 서버 문을 엽니다.
4. Fly.io 원격 서버에서 새로운 도커 박스를 빌드하고, 무중단으로 기존 서버와 교체합니다.

## 7. 🔒 보안 및 확장성 아키텍처 (앞으로의 과제)

현재는 앱 주소(`taskflow-todo.fly.dev`)만 알면 누구나 들어와서 데이터를 조작할 수 있습니다. 이를 방지하기 위한 **3중 보안 시스템**이 예정되어 있습니다.

### 1단계: Google OAuth (소셜 로그인 인증)
앱 접속 시 구글 로그인 팝업을 띄워 진짜 '사람'이고 구글 계정이 있는 사용자만 1차적으로 통과시킵니다. 봇(Bot)이나 크롤러는 로그인 팝업을 뚫지 못해 자동으로 차단됩니다.

### 2단계: 이메일 화이트리스트 (Email Allowlist) 인가 시스템
구글 로그인을 통과했더라도 아무나 쓰면 안 됩니다.
데이터베이스에 `AllowedEmails` 테이블을 만들고, 관리자가 미리 등록해 둔 이메일(예: 본인 이메일)과 일치하는 사용자만 앱 화면에 진입할 수 있도록 백엔드 로직(Authorization)을 추가합니다. 승인되지 않은 유저는 "접근 권한 없음" 화면에서 튕겨나갑니다.

### 3단계: API Rate Limiting (디도스/스팸 방어)
로그인을 뚫은 해커나 악성 스크립트가 1초에 할 일을 10,000개씩 등록해서 서버를 마비시키는 것을 막아야 합니다.
ASP.NET Core의 강력한 **Rate Limiter 미들웨어**를 추가하여, "같은 IP 또는 같은 유저가 1초에 10번 이상 요청하면 1분간 차단(HTTP 429 Too Many Requests)" 하도록 방어막을 씌울 예정입니다.
